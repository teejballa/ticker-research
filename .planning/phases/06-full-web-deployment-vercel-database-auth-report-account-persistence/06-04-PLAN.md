Vercel build command for Prisma (from RESEARCH.md Pitfall 1):
```json
{ "buildCommand": "prisma migrate deploy && next build" }
```

Required env vars for web mode:
- GOOGLE_CLIENT_ID: Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client ID
- GOOGLE_CLIENT_SECRET: same location as Client ID
- NEXTAUTH_SECRET: generate with `openssl rand -base64 32`
- NEXTAUTH_URL: your Vercel deployment URL (e.g., https://your-app.vercel.app)
- DATABASE_URL: Neon pooled connection string (from Neon dashboard, has `-pooler` in hostname)
- DIRECT_URL: Neon direct connection string (from Neon dashboard, no `-pooler`)
- DEPLOYMENT_MODE: set to `web` in Vercel environment variables

Existing env vars already documented (from Phase 4):
- ANTHROPIC_API_KEY: required for all modes
- DAYTONA_CONTAINER_URL: required for cloud/web mode analysis

Prisma migration workflow:
- `npx prisma migrate dev --name init` generates the SQL migration file from schema.prisma
- This requires DATABASE_URL to be set (run with Neon direct URL)
- The generated `prisma/migrations/` directory MUST be committed to the repo
- `prisma migrate deploy` (run on Vercel build) applies existing migration files — it does NOT generate new ones
- If no migration files exist in the repo, `prisma migrate deploy` is a no-op and the DB table is never created

**Step 1 — Generate the initial Prisma migration.**

This step requires `DATABASE_URL` and `DIRECT_URL` to be set. The user must have provisioned a Neon project and have these connection strings available. If they are not set yet, the executor should prompt the user to set them before continuing.

Check if DATABASE_URL is set:
```bash
[ -z "$DATABASE_URL" ] && echo "DATABASE_URL not set — set it in .env.local before running this step" || echo "DATABASE_URL is set"
```

If DATABASE_URL is set, run:
```bash
cd /Users/tj/Desktop/Ticker-Research
npx prisma migrate dev --name init
```

This command will:
1. Create the `prisma/migrations/` directory
2. Generate a timestamped migration file (e.g., `prisma/migrations/20260320000000_init/migration.sql`) containing the `CREATE TABLE "reports"` SQL
3. Apply the migration to the Neon database

**Critical:** Commit the generated `prisma/migrations/` directory. Without it, `prisma migrate deploy` on Vercel has nothing to apply and the `reports` table will never be created.

```bash
git add prisma/migrations/
git status
```

If DATABASE_URL is NOT available yet (user hasn't provisioned Neon), use `prisma db push` as a LOCAL DEVELOPMENT fallback ONLY:
```bash
# DEV FALLBACK ONLY — do NOT use in production or change vercel.json to use this.
# prisma db push applies the schema directly without creating a migration file.
# Since no migration file is generated, prisma migrate deploy on Vercel will still be
# a no-op and the DB table will NOT exist in production.
# You MUST return and run 'npx prisma migrate dev --name init' with a real Neon DATABASE_URL
# before deploying to Vercel. vercel.json buildCommand must always be 'prisma migrate deploy && next build'.
npx prisma db push
```

**Production path (ALWAYS use for Vercel deployments):**
- Run `npx prisma migrate dev --name init` with the real Neon DATABASE_URL.
- Commit `prisma/migrations/`.
- vercel.json buildCommand stays as `prisma migrate deploy && next build`.
