Create the `.devcontainer/` directory and write `.devcontainer/devcontainer.json` using the pattern from RESEARCH.md §Pattern 3.

Required content:
```json
{
  "name": "ticker-research",
  "image": "mcr.microsoft.com/devcontainers/python:3.12",
  "features": {
    "ghcr.io/devcontainers/features/node:1": { "version": "18" }
  },
  "postCreateCommand": "pip install -r scripts/requirements.txt && playwright install --with-deps chromium && npm install",
  "forwardPorts": [3000, 8080],
  "remoteEnv": {
    "ANTHROPIC_API_KEY": "${localEnv:ANTHROPIC_API_KEY}"
  }
}
```

Critical implementation notes (from RESEARCH.md §Pitfall 3):
- Use `playwright install --with-deps chromium` NOT `playwright install chromium`. The `--with-deps` flag installs required glibc OS-level system libraries (libnss3, libgbm, etc.) that Playwright requires on Linux. Without it, Chromium fails to launch inside the container with missing shared library errors.
- The base image `mcr.microsoft.com/devcontainers/python:3.12` is Ubuntu-based (glibc). Do NOT use Alpine-based images — Playwright does not support musl libc.
- `npm install` is included in `postCreateCommand` so the Next.js server can be started inside the container with `npm start` (or `npm run dev`).
- Auth is NOT in the image. After `daytona create`, the user runs `notebooklm login` once inside the container; credentials persist at `~/.notebooklm/storage_state.json` across container restarts.
- `forwardPorts: [3000, 8080]` exposes the Next.js server (3000) and an optional alternate port (8080). Daytona generates a stable public URL for forwarded ports that becomes the `DAYTONA_CONTAINER_URL` value on Vercel.
- `remoteEnv.ANTHROPIC_API_KEY` maps the local machine's env var into the container so `npm start` (via prestart → setup.sh) passes the API key check. Users who set `ANTHROPIC_API_KEY` in their shell automatically have it available inside the container.

Add a comment block above the JSON explaining the one-time user steps after `daytona create`:

```
// 3. Auth persists at ~/.notebooklm/storage_state.json across restarts
// 4. Set DAYTONA_CONTAINER_URL on Vercel to the forwarded port 3000 URL
// 5. Start the server: npm start
```

<task type="auto">
  <name>Task 2: vercel.json — SSE maxDuration configuration</name>
  <files>vercel.json</files>
  <action>
Create `vercel.json` at the project root with maxDuration set on the analysis and research API routes. Use the template from RESEARCH.md §Code Examples (vercel.json section):

```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "functions": {
    "src/app/api/analysis/**/*": {
      "maxDuration": 300
    },
    "src/app/api/research/**/*": {
      "maxDuration": 300
    }
  }
}
```

Why 300: The Vercel Hobby plan with Fluid Compute enabled allows up to 300s (5 minutes). A notebooklm-py research run takes ~3–5 minutes (text index sleep + URL index sleep + 6 async queries). This is the maximum available on Hobby. RESEARCH.md §Pitfall 4 notes that Pro plan (800s) provides more headroom for slow conditions — document this in a comment in the file using a `"//comment"` workaround if desired, but since JSON doesn't support comments, add the note to `.devcontainer/README.md` or a project wiki instead.

Do NOT set maxDuration on other routes — only the long-running SSE proxy routes need it. Setting a high maxDuration globally would cause other fast routes (setup/status, ticker search) to delay Vercel's cold-start optimization.
