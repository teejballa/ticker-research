## Reddit Auth + Client

### Q1: Which Reddit OAuth flow should we use?

| Option | Description | Selected |
|--------|-------------|----------|
| App-only OAuth (Recommended) | Client credentials flow. REDDIT_CLIENT_ID + REDDIT_CLIENT_SECRET only. Read-only public content. 2 env vars vs 4. No user password. 100 QPM. Token TTL 24h cached in Upstash. | ✓ |
| Script-type OAuth (full) | Password grant. + REDDIT_USERNAME + REDDIT_PASSWORD. Slightly more access but Cipher doesn't need it. 100 QPM same. | |
| Unauthenticated old.reddit.com .json | Append `.json`. No OAuth. But Reddit blocks unauthenticated by IP — same anti-pattern as Yahoo throttle. | |

**User's choice:** App-only OAuth (Recommended)

### Q2: Library / client choice?

| Option | Description | Selected |
|--------|-------------|----------|
| Raw fetch + Upstash token cache (Recommended) | Zero deps. ~30 LOC. Matches apewisdom.ts pattern. | ✓ |
| snoowrap npm package | TS wrapper, maintenance-mode (last release 2022). Extra deps. | |
| reddit-axios / @devvit clients | Newer alternatives. Less proven for serverless. | |

**User's choice:** Raw fetch + Upstash token cache (Recommended)

### Q3: Token cache + rate-limit strategy in Upstash?

| Option | Description | Selected |
|--------|-------------|----------|
| Single shared token + token-bucket (Recommended) | `reddit:oauth_token` TTL=expires_in-60s with SETNX guard; `reddit:rate_bucket` atomic Lua INCR/DECR. Survives cold starts. | ✓ |
| Per-lambda in-memory token | Each lambda fetches own token. Wastes quota + Phase 30 D-04 already rejected this pattern. | |
| Token in process.env at deploy | Pre-fetch + set env. Brittle 24h expiry. | |

**User's choice:** Single shared token + token-bucket (Recommended)

### Q4: 429 handling mid-scan?

| Option | Description | Selected |
|--------|-------------|----------|
| Trip breaker + skip ticker (Recommended) | 429 → RATE_LIMITED error_class → withBreaker D-05 trip → BreakerOpenError short-circuits → ticker_skipped++. Matches "crons never 500". | ✓ |
| Backoff + retry inline | 100 QPM = 600ms spacing; retries chew the cron window. Worse than skipping. | |
| Pre-emptive throttle only (no breaker on 429) | Token bucket alone. No defense if Reddit changes limits. | |

**User's choice:** Trip breaker + skip ticker (Recommended)

## Cross-Cutting: Report-Generation Path

### Q10: How should report-generation use the new community-scan?

| Option | Description | Selected |
|--------|-------------|----------|
| Same call site, parallel fan-out (Recommended) | `lightweightCommunityScan(ticker)` unchanged signature. 20 sub + 1 HN call in Promise.all. ~3-5s wall time. Shared Upstash bucket with cron. | ✓ |
| Cached SentimentSnapshot if fresh (<1h) | Reuse recently-scanned data. Long-tail tickers always cache-miss. | |
| Two-tier: hot tickers cache, cold tickers live scan | Hybrid. Adds telemetry + freshness threshold complexity. | |

**User's choice:** Same call site, parallel fan-out (Recommended)

### Q11: Reddit QPM budget allocation between cron + report paths?

| Option | Description | Selected |
|--------|-------------|----------|
| Shared bucket, report path priority (Recommended) | Single 100 QPM bucket. Report claims first. Cron throttles when bucket < 30. | ✓ |
| Hard split: 70 QPM cron / 30 QPM reports | Two buckets keyed by caller. Under-utilizes ceiling when one path idle. | |
| FIFO single bucket, no priority | Simplest. But busy cron starves clicking user. | |

**User's choice:** Shared bucket, report path priority (Recommended)

## Data Shape + Adapter Architecture

### Q12: Data shape — Reddit JSON vs markdown regex?

| Option | Description | Selected |
|--------|-------------|----------|
