# Phase 30.1 Operator Playbook — Xpoz Cutover → Firecrawl Removal

**Audience:** Operator (TJ) only. Spans ~3-5 calendar days.

**Updated 2026-05-16:** Plan 30.1 pivoted from Reddit OAuth to Xpoz Pro after
Reddit closed self-service API access via the Responsible Builder Policy
(2025). The original 6-stage Reddit-OAuth runbook is replaced with a 3-stage
Xpoz cutover; the Firecrawl-removal stages remain. See `30.1-CONTEXT.md
§"2026-05-16 Amendment — Xpoz Pivot"` for the full rationale and decisions
D-31..D-40.

This playbook drives the Phase 30.1 community-scan migration end-to-end:
verify Xpoz setup → 24h shadow soak → cutover → resolve the open Firecrawl
`ProviderHealthAlert` → human UAT → land the final Firecrawl-removal commit.

See also:
- `.planning/phases/30.1-free-community-scan-migration/30.1-CONTEXT.md` —
  D-31..D-40 (pivot decisions) and the cost projection (~\$67/mo).
- `.planning/phases/30.1-free-community-scan-migration/30.1-HUMAN-UAT.md` —
  done-gate SQL + golden-ticker UAT.

## Stage 1: Verify Xpoz Setup (Day 0, ~5 min)

Confirm `XPOZ_API_KEY` is set in Vercel Production + Development environments:

```bash
vercel env ls | grep XPOZ_API_KEY
```

Expected output: two rows, one for `production` and one for `development`,
both showing `Encrypted`. If either is missing:

```bash
vercel env add XPOZ_API_KEY production
vercel env add XPOZ_API_KEY development
```

Paste the key (starts with `xpoz_...`) when prompted. Trigger a redeploy
after adding so the new env var lands in the running lambdas.

Local sanity check (optional):

```bash
printenv XPOZ_API_KEY | head -c 10
```

Should print the first 10 characters, NOT empty.

## Stage 2: 24h Shadow Soak (Day 0 → Day 1)

Flip `FEATURE_COMMUNITY_SCAN_SOURCE` from `firecrawl` to `shadow`:

```bash
# Via dashboard: Project → Settings → Environment Variables → Production
# Or via CLI:
vercel env rm FEATURE_COMMUNITY_SCAN_SOURCE production
vercel env add FEATURE_COMMUNITY_SCAN_SOURCE production
# Enter value: shadow
```

Trigger a production redeploy (push a no-op commit or use the "Redeploy"
button on the latest production deployment).

Within 1 hour, verify Xpoz calls START appearing in `ProviderCallLog`:

```sql
SELECT provider_id, status, COUNT(*) FROM "ProviderCallLog"
 WHERE started_at > NOW() - INTERVAL '1 hour'
   AND provider_id IN ('reddit-xpoz', 'twitter-xpoz')
 GROUP BY 1, 2;
```

Expected: `reddit-xpoz` count > 0, `twitter-xpoz` count > 0, status mostly `ok`.

Verify Firecrawl calls STILL FIRE (legacy path stays canonical in shadow mode
— the new path runs in the background via `after()`):

```sql
SELECT COUNT(*) FROM "ProviderCallLog"
 WHERE started_at > NOW() - INTERVAL '1 hour'
   AND provider_id = 'firecrawl';
```

After 24h, run the error-rate probe:

```sql
SELECT provider_id,
       COUNT(*) AS total,
       SUM(CASE WHEN status='error' THEN 1 ELSE 0 END)::float / COUNT(*) AS error_rate
  FROM "ProviderCallLog"
 WHERE provider_id IN ('reddit-xpoz', 'twitter-xpoz')
   AND started_at > NOW() - INTERVAL '24 hours'
 GROUP BY 1;
```

**Pass criteria:** both error rates < 10%.

**Kill criteria — flip back to `firecrawl` and investigate:**

| Trigger | Condition | Action |
|---|---|---|
| Xpoz error spike | error_rate > 25% for > 4h on either provider_id | Flip `FEATURE_COMMUNITY_SCAN_SOURCE` back to `firecrawl`, open an issue. |
| Auth errors | Any rows with `error_class = 'AUTH_FAILED'` in the last 1h | Re-verify `XPOZ_API_KEY` in Vercel; rotate if compromised. |
| Zero traffic | After 1h, both reddit-xpoz and twitter-xpoz counts are 0 | Verify the redeploy succeeded and the env flip landed; check `/api/cron/sentiment-scan` cron is firing on schedule. |

## Stage 3: Cutover (Day 1 once Stage 2 is GREEN)

Flip `FEATURE_COMMUNITY_SCAN_SOURCE` from `shadow` to `xpoz`:

```bash
vercel env rm FEATURE_COMMUNITY_SCAN_SOURCE production
vercel env add FEATURE_COMMUNITY_SCAN_SOURCE production
# Enter value: xpoz
```

Redeploy. Within 30 minutes, verify Firecrawl call volume drops to zero:

```sql
SELECT COUNT(*) FROM "ProviderCallLog"
 WHERE started_at > NOW() - INTERVAL '30 minutes'
   AND provider_id = 'firecrawl';
```

Expected: 0.

Wait 24h, then run the **golden-ticker UAT** for AAPL, NVDA, GME, TSLA:

```bash
# For each ticker, trigger a fresh research request and inspect the
# Community Intelligence section in the rendered report.
curl -X POST https://ciphersearch.app/api/research/AAPL
```

**Each report's Community Intelligence section must show:**
- ≥3 highlights with non-empty subreddit / Twitter / HackerNews community names
- At least one clickable Reddit permalink (`https://www.reddit.com/r/.../comments/...`)
- At least one clickable Twitter URL (`https://twitter.com/.../status/...`) when authentic posts exist
- Non-zero `engagement_count` values

Capture results in `30.1-HUMAN-UAT.md`.

## Stage 4: Done-Gate Verification (Day 2)

Run all four D-28 SQL checks:

```sql
-- (1) Reddit-Xpoz error rate over 24h
SELECT COUNT(*) AS total,
       SUM(CASE WHEN status='error' THEN 1 ELSE 0 END)::float / COUNT(*) AS error_rate
  FROM "ProviderCallLog"
 WHERE provider_id = 'reddit-xpoz'
   AND started_at > NOW() - INTERVAL '24 hours';
-- PASS when total >= 50 AND error_rate < 0.10

-- (2) Twitter-Xpoz error rate over 24h
SELECT COUNT(*) AS total,
       SUM(CASE WHEN status='error' THEN 1 ELSE 0 END)::float / COUNT(*) AS error_rate
  FROM "ProviderCallLog"
 WHERE provider_id = 'twitter-xpoz'
   AND started_at > NOW() - INTERVAL '24 hours';
-- PASS when total >= 50 AND error_rate < 0.10

-- (3) Firecrawl call volume = 0 over 24h
SELECT COUNT(*) FROM "ProviderCallLog"
 WHERE provider_id = 'firecrawl'
   AND started_at > NOW() - INTERVAL '24 hours';
-- PASS when count = 0

-- (4) Firecrawl ProviderHealthAlert resolved
SELECT COUNT(*) FROM "ProviderHealthAlert"
 WHERE provider_id = 'firecrawl' AND resolved_at IS NULL;
-- PASS when count = 0
```

Force-resolve the Firecrawl alert via the migration script shipped in
Plan 30.1-05 Task 3:

```bash
npx tsx scripts/migrations/30.1-resolve-firecrawl-alert.ts
```

Idempotent — safe to re-run.

## Stage 5: Firecrawl Removal (Day 3+)

Once all done-gate SQL queries PASS and the golden-ticker UAT is signed off,
land the final Firecrawl-removal commit. This deletes:

- `@mendable/firecrawl-js` from `package.json` + lockfile
- The `firecrawl` branch in `src/lib/data/lightweight-community-scan.ts`
  (including `runFirecrawlPath` and the `FEATURE_COMMUNITY_SCAN_SOURCE`
  flag itself)
- `'firecrawl'` from `FieldOrigin`, `source-class`, `cost-estimators`,
  `check-telemetry-coverage` whitelists
- All `FIRECRAWL_API_KEY` references across the codebase

The original Plan 30.1-05 Task 5 enumeration of files-to-touch still applies.

After the commit lands and CI is green, the `FEATURE_COMMUNITY_SCAN_SOURCE`
env var can be deleted from Vercel — the flag itself is gone from code, so
reading it would be a no-op.

## Rollback

If anything breaks at any stage **before Stage 5**:

1. Flip `FEATURE_COMMUNITY_SCAN_SOURCE` back to `firecrawl` in Vercel.
2. Redeploy.
3. The legacy Firecrawl path is preserved untouched until Stage 5 commits;
   rollback is a single env flip + redeploy.

If something breaks **after Stage 5** (Firecrawl removal commit landed), the
rollback is a `git revert` of the removal commit + redeploy. There is no
env-only undo because the `COMMUNITY_SCAN_SOURCE` flag is gone.
