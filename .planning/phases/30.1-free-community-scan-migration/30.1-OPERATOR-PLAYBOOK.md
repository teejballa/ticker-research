## Stage 2: Soak (Day 0 â†’ Day 7)

Let shadow mode run for 7 calendar days. **DO NOT INTERVENE** unless a kill criterion fires:

| Trigger | Condition | Action |
|---|---|---|
| Reddit error spike | `error_rate > 25%` for > 4h consecutive (per `/insights/provider-health` or the SQL probe below) | Flip `FEATURE_COMMUNITY_SCAN_SOURCE` back to `firecrawl`, open an issue, investigate before retrying. |
| Zero observations | `SELECT COUNT(*) FROM "SentimentObservation" WHERE source IN ('reddit', 'hackernews') AND fetched_at > NOW() - INTERVAL '24 hours'` < 50 after 24h | Re-check env vars (`REDDIT_CLIENT_ID`, `REDDIT_CLIENT_SECRET`, `SENTIMENT_AUTHOR_PEPPER`). Verify the `/api/cron/sentiment-scan` cron is running on schedule. |
| HackerNews 5xx spike | `error_rate > 25%` for > 4h on the `hackernews` provider | Algolia HN search is rate-limit-free and almost never throws â€” investigate any spike as a code regression. |

Error-rate probe SQL (run during the soak as needed):

```sql
SELECT provider_id,
       COUNT(*) AS total,
       SUM(CASE WHEN status='error' THEN 1 ELSE 0 END)::float / COUNT(*) AS error_rate
  FROM "ProviderCallLog"
 WHERE provider_id IN ('reddit', 'hackernews')
   AND started_at > NOW() - INTERVAL '4 hours'
 GROUP BY 1;
```

### Concurrent-ticker cap (D-12)

`src/lib/ticker-watchlist.ts` currently dispatches up to 20 tickers per cron run. RESEARCH.md Â§"Concurrent-Ticker Cap" derives the cap from the upstream rate limit and per-call latency budget:

- Reddit OAuth allows 60 req/min per client_id; each ticker scan issues â‰¤4 subreddit fetches (1 per `COMMUNITY_SUBS` entry). Worst case: 80 req/min â€” over budget at the start of a cron tick before the per-second smoothing window settles.
- Halve to 30/min worst-case â‡’ 15 tickers/cron is the safe ceiling.

Operator check during Stage 2: if the cron's observed p95 latency-per-call (from `ProviderCallLog.duration_ms` for `provider_id='reddit'`) exceeds 300 ms, drop the per-cron-run cap from 20 â†’ 15 tickers in `src/lib/ticker-watchlist.ts`. If p95 < 300 ms, the 20-ticker default holds â€” no change needed (D-12 closed-as-default).

## Rollback

If anything breaks at any stage **before Stage 6**:

1. In Vercel production env, flip `FEATURE_COMMUNITY_SCAN_SOURCE` back to `firecrawl`.
2. Redeploy.
3. The legacy Firecrawl path is preserved untouched until Stage 6 commits; rollback is a single env flip + redeploy.

If something breaks **after Stage 6** (Firecrawl removal commit landed), the rollback is a `git revert` of the removal commit + redeploy. There is no env-only undo because the `COMMUNITY_SCAN_SOURCE` flag is gone.
