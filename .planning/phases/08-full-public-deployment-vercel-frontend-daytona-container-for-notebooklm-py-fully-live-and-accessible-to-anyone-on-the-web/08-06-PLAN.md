<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 1: Provision Daytona workspace + start FastAPI server</name>
  <what-built>
    Daytona workspace creation and FastAPI server startup — these require infrastructure access (Daytona account, container provisioning) and live environment credentials that Claude cannot execute on behalf of the user.
  </what-built>
  <how-to-verify>
    Follow these steps to provision and verify:

    STEP 1 — Install Daytona CLI if not present:
    ```bash
    curl -sf -L https://download.daytona.io/daytona/install.sh | sh
    daytona --version
    ```

    STEP 2 — Create Daytona workspace from devcontainer.json.
    The workspace must be persistent (never auto-stop) and publicly accessible.
    Using the Daytona Python SDK (DAYTONA_API_KEY must be set):
    ```python
    # Run: python3 scripts/create_daytona_workspace.py (one-time)
    from daytona import Daytona, CreateSandboxFromSnapshotParams
    daytona = Daytona()
    sandbox = daytona.create(CreateSandboxFromSnapshotParams(
        ephemeral=False,
        auto_stop_interval=0,  # never auto-stop
        public=True,            # preview URLs publicly accessible without Daytona auth
    ))
    port_8080_url = sandbox.get_preview_link(8080)
    print(f"Container URL: {port_8080_url}")
    ```
    Record the workspace ID and the port-8080 preview URL.

    STEP 3 — SSH into the workspace and start the FastAPI server:
    ```bash
    daytona ssh ticker-research  # or use Daytona web terminal
    # Inside container:
    cd /workspaces/ticker-research
    pip install -r scripts/requirements.txt
    export DAYTONA_SECRET=<generate-a-random-secret>
    export ALLOWED_ORIGIN=https://ticker-research.vercel.app
    nohup python3 scripts/container_server.py &
    # Verify locally:
    curl http://localhost:8080/health
    ```

    STEP 4 — Set DAYTONA_SECRET and ALLOWED_ORIGIN persistently in the container:
    ```bash
    echo "export DAYTONA_SECRET=<your-secret>" >> ~/.bashrc
    echo "export ALLOWED_ORIGIN=https://ticker-research.vercel.app" >> ~/.bashrc
    ```

    STEP 5 — Verify the health endpoint is reachable from outside the container:
    ```bash
    curl -sf https://8080-<SANDBOX_ID>.proxy.daytona.works/health
    # Expected: {"status":"ok"}
    ```

    Confirm all of the following before resuming:
    1. Daytona workspace is running
    2. FastAPI server is running on port 8080 inside the container
    3. GET https://8080-{SANDBOX_ID}.proxy.daytona.works/health returns {"status": "ok"}
    4. DAYTONA_SECRET and ALLOWED_ORIGIN are set in the container environment
    5. Container URL is recorded for use in the next task
  </how-to-verify>
  <resume-signal>Type "daytona ready: {container-url}" with the full container URL (e.g., "daytona ready: https://8080-abc123.proxy.daytona.works"), or describe what failed</resume-signal>
</task>

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Configure Vercel env vars + deploy</name>
  <what-built>
    Vercel environment configuration, Neon DB migration, Google OAuth production setup, and Vercel deployment — these require dashboard access and live credentials that Claude cannot execute.
  </what-built>
  <how-to-verify>
    STEP 1 — Run Neon production DB migration.
    If DATABASE_URL is already set in .env.local for the production Neon instance:
    ```bash
    npx prisma migrate deploy
    ```
    Confirm the migration succeeded:
    ```bash
    npx prisma db execute --stdin <<< "SELECT COUNT(*) FROM user_credentials;"
    ```

    STEP 2 — Add production redirect URI to Google Cloud Console.
    Navigate to: console.cloud.google.com → APIs & Services → Credentials → OAuth 2.0 Client ID
    Add to Authorized redirect URIs: `https://ticker-research.vercel.app/api/auth/callback/google`
    (Use the actual Vercel domain once known — Vercel assigns it on first deploy.)

    STEP 3 — Set all 11 Vercel env vars in the Vercel project settings dashboard (or via CLI):
    ```
    DATABASE_URL=<neon-connection-string>
    NEXTAUTH_SECRET=<openssl rand -hex 32>
    NEXTAUTH_URL=https://ticker-research.vercel.app
    GOOGLE_CLIENT_ID=<from-google-cloud-console>
    GOOGLE_CLIENT_SECRET=<from-google-cloud-console>
    ANTHROPIC_API_KEY=<anthropic-key>
    DEPLOYMENT_MODE=web
    NEXT_PUBLIC_DEPLOYMENT_MODE=web
    DAYTONA_CONTAINER_URL=https://8080-<SANDBOX_ID>.proxy.daytona.works
    DAYTONA_SECRET=<same-secret-set-in-container>
    CREDENTIAL_ENCRYPTION_KEY=<openssl rand -hex 32>
    ```
    Via CLI (example):
    ```bash
    vercel env add DEPLOYMENT_MODE production
    vercel env add NEXT_PUBLIC_DEPLOYMENT_MODE production
    vercel env add DAYTONA_CONTAINER_URL production
    vercel env add DAYTONA_SECRET production
    vercel env add CREDENTIAL_ENCRYPTION_KEY production
    ```

    STEP 4 — Deploy to Vercel:
    ```bash
    vercel --prod
    ```
    OR push to main branch if Vercel is connected to GitHub:
    ```bash
    git push origin main
    ```

    STEP 5 — Verify build succeeds in Vercel dashboard. Check build logs for any env var errors or TypeScript failures.

    Confirm all of the following before resuming:
    1. Neon production DB has user_credentials table (prisma migrate deploy succeeded)
    2. All 11 env vars are set in Vercel project settings
    3. Google OAuth has production redirect URI configured
    4. Vercel deployment succeeded (green build)
    5. App is accessible at https://ticker-research.vercel.app (or equivalent)
  </how-to-verify>
  <resume-signal>Type "vercel deployed: {url}" with the live Vercel URL, or describe what failed</resume-signal>
