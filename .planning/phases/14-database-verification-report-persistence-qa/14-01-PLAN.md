## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API route → Neon DB | `user_id` from NextAuth session scopes all queries — client cannot choose another user's data |
| `[filename]` URL param → readReportFromDb | Param treated as opaque UUID string; Prisma findFirst with user_id guard prevents enumeration |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Information Disclosure | `readReportFromDb(id, userId)` | mitigate | `findFirst({ where: { id, user_id } })` — DB-level scoping already implemented; unit test (DB-QA-06) verifies throws on mismatch |
| T-14-02 | Tampering | `[filename]` route param in web mode | accept | UUID is an opaque string; Prisma returns null for non-existent IDs; no path traversal risk in web mode (no filesystem access) |
| T-14-03 | Information Disclosure | `listReportsFromDb` now returns `id` field | accept | `id` is scoped to the authenticated user via `where: { user_id }` — returning the UUID to an authenticated user is correct behavior; UUID alone is insufficient for cross-user access |
| T-14-04 | Spoofing | NextAuth session in API routes | mitigate | All web-mode DB operations gate on `getServerSession(authOptions)` returning a non-null session with user.email |
</threat_model>
